# 变更控制流程

## 目标

管理项目范围、需求、架构、数据契约、评分算法、安全策略、发布策略等重大变更，确保任何关键变更可追踪、可评审、可批准。

## 需要变更控制的事项

- 一期范围变化。
- 非范围功能进入一期。
- 技术路线变化。
- 架构重大调整。
- schema 破坏性变更。
- CLI 命令语义变化。
- 报告格式变化。
- 安全风险接受。
- 默认开启高风险能力。
- 删除数据策略变化。
- 发布策略变化。

## 标准步骤

1. 创建 Change Request。
2. 描述变更原因和目标。
3. 分析影响范围。
4. 关联需求、设计、任务和测试。
5. 指定需要介入的 Agent。
6. 生成可选方案。
7. 评估风险和成本。
8. 生成 Decision Pack。
9. 由人类批准或驳回。
10. 更新 Project Pack、ADR、schema、任务包或测试。

## 输出

```text
change-request.yaml
decision-pack.md
ADR 或更新说明
影响分析
测试补充计划
```

## 准出条件

- 变更有明确原因。
- 影响范围已分析。
- 必须人类批准的事项已批准。
- 相关文档和测试已更新。
